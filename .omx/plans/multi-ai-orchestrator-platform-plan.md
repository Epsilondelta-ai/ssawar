# 멀티 AI 오케스트레이터 플랫폼 기획안

작성일: 2025-02-14
상태: Draft v1

## Brand / Product Split

### 브랜드명
- `ssawar`
- 한글 발음은 `싸워`

### 대외 인식
- 유저는 `AI들끼리 싸움을 붙이는 서비스`로 이해하게 만든다.
- 도메인은 `ssawar.ai`

### 내부 실체
- 실제 제품 코어는 `범용 멀티 AI 오케스트레이터`
- 배틀은 첫 번째이자 가장 공격적인 시장 진입 경험

### 핵심 원칙
- 브랜드는 좁게: `싸움`
- 엔진은 넓게: `오케스트레이션`

## Core Thesis

이 서비스의 본질은 `AI 배틀 서비스` 자체가 아니라, 여러 AI를 한 세션 안에서 특정 규칙에 따라 움직이게 만드는 `멀티 AI 오케스트레이터`다.

즉,
- `배틀`은 첫 번째 사용자 경험
- `오케스트레이터`는 장기적으로 남는 핵심 자산

이 구조를 택하면 제품은 하나의 강한 진입점으로 시작하면서도, 내부 엔진은 게임, 토론, 경쟁, 협업, 취합 같은 다양한 모드로 확장할 수 있다.

## Strategic Positioning

### 대외 포지셔닝
- 초기 출시 메시지는 단순해야 한다.
- 따라서 1차 출시에서는 `ssawar = AI 배틀`만 전면에 내세운다.
- 일반 사용자가 한 번에 이해할 수 있는 경험은 배틀이 가장 강하다.

### 내부 포지셔닝
- 내부적으로는 `mode-specific app`가 아니라 `orchestration platform`으로 설계한다.
- 배틀, 마피아, 끝말잇기, 아이디어 회의는 전부 다른 서비스가 아니라 같은 엔진의 다른 규칙셋이다.

## Why This Matters

### 배틀만 하드코딩하면 생기는 문제
- 새 모드를 추가할 때마다 로직을 거의 다시 짜야 한다.
- 턴 제어, 참가자 선택, 상태 전환, 종료 조건이 모드마다 중복 구현된다.
- 제품이 커질수록 규칙 충돌과 유지보수 비용이 커진다.

### 오케스트레이터로 추상화하면 생기는 장점
- 새로운 모드는 `규칙 템플릿`만 추가해도 붙일 수 있다.
- 공통 로직은 재사용한다.
- 모델 공급자가 바뀌어도 모드 로직은 유지된다.
- 관전자 UX, 로그 저장, 공유, 모더레이션 같은 횡단 기능이 공통화된다.

## Orchestrator Definition

오케스트레이터 AI 또는 오케스트레이터 시스템의 역할은 다음과 같다.

1. 세션 타입 결정
2. 참가자 역할 부여
3. 턴 순서 또는 동시 실행 규칙 결정
4. 현재 페이즈 전환
5. 각 참가자에게 맞는 프롬프트 생성
6. 결과 수집 및 정규화
7. 진행 지속 여부 결정
8. 점수, 승패, 요약 또는 산출물 생성

중요한 점은, 오케스트레이터는 반드시 하나의 LLM일 필요가 없다는 것이다.
- 일부는 규칙 기반 상태머신
- 일부는 LLM 기반 판단
- 일부는 혼합형 정책

MVP에서는 비용과 예측 가능성을 위해 `규칙 기반 + 필요한 곳에만 LLM 보조판단`이 적합하다.

### Orchestrator Model Selection
- 어떤 오케스트레이터 모델을 쓸지는 유저가 `채팅 세션 생성 시점`에 선택한다.
- 오케스트레이터 모델과 참가 AI 목록은 별도 필드로 관리한다.
- 오케스트레이터는 플레이어가 아니라 상위 진행자다.

## Core Engine Abstractions

### 1. Session
- 하나의 멀티 AI 상호작용 단위
- 예: 배틀 한 판, 마피아 한 게임, 끝말잇기 한 판, 브레인스토밍 세션

### 2. Participants
- AI 모델들
- 필요 시 인간 사용자도 참가자로 포함 가능
- 각 참가자는 `모델`, `닉네임`, `역할`, `행동권한`을 가진다.
- 참가 AI 최대 수는 16으로 둔다.

### 2-1. Orchestrator Slot
- 세션마다 정확히 1개의 오케스트레이터 슬롯이 있다.
- 오케스트레이터는 참가자 배열과 분리된 제어자다.
- 참가자와 같은 모델 계열을 오케스트레이터로 다시 써도 된다.

### 3. Roles
- 예: 사회자, 공격자, 수비자, 시민, 마피아, 검토자, 아이디어 발산자, 요약자
- 같은 모델도 역할만 다르게 주면 전혀 다른 행동을 하게 만들 수 있다.

### 4. Phase
- 세션의 현재 국면
- 예: 준비, 오프닝, 토론, 투표, 밤, 낮, 요약, 종료

### 5. Turn Policy
- 누가 언제 말하는지 정의
- 종류:
- 순차 턴
- 동시 응답
- 지목 반응
- 역할 기반 우선권
- 유저 개입 기반 호출

### 6. Action Type
- attack
- defend
- accuse
- vote
- answer
- summarize
- propose
- eliminate
- reveal

### 7. Stop Policy
- 최대 턴 수
- 점수 수렴
- 반복 감지
- 승패 확정
- 유저 종료
- 시간 초과

### 8. Outcome Resolver
- 승자 계산
- 생존자 계산
- 정답 정확도 계산
- 아이디어 취합
- 요약 생성

## Execution Patterns

### Pattern A. 순차 턴
- 끝말잇기
- 발언 순서가 중요한 토론
- 역할극

### Pattern B. 동시 응답
- 틀린그림찾기
- 정답 맞히기
- 같은 질문에 대한 병렬 답변 비교

### Pattern C. 페이즈 기반 게임
- 마피아
- 추리 게임
- 협상 게임

### Pattern D. 혼합형
- 먼저 각자 답변
- 이후 서로 토론
- 마지막에 요약 또는 투표

## Mode Templates

### 1. Battle Mode
- 목적: 누가 더 설득력 있는지, 더 웃긴지, 더 논리적인지 경쟁
- 핵심 규칙: 반박, 재반박, 동적 연장, 요약 종료
- 핵심 출력: 하이라이트, 요약 카드, 세션 로그

### 2. Mafia Mode
- 목적: 역할 기반 사회적 추리 게임
- 오케스트레이터 역할:
- 역할 배분
- 밤/낮 전환
- 제거 판정
- 투표 집계
- 공개 정보와 비공개 정보 분리
- 핵심 난점: 참가자별 비밀 상태 관리

### 3. Spot-The-Difference Mode
- 목적: 누가 더 빠르고 정확히 차이를 찾는지 경쟁
- 오케스트레이터 역할:
- 모든 참가자에게 동시에 동일 입력 제공
- 응답 시간 측정
- 정답 검증
- 점수 계산
- 핵심 난점: 정답 판정 기준의 엄격성

### 4. Word-Chain Mode
- 목적: 정해진 순서로 언어 게임 진행
- 오케스트레이터 역할:
- 시작 단어 제시
- 순서 강제
- 유효 단어 검증
- 탈락 처리
- 핵심 난점: 사전 검증과 시간 제한

### 5. Brainstorm Mode
- 목적: 질문에 대해 다양한 관점의 아이디어를 생산하고 취합
- 오케스트레이터 역할:
- 유저에게 진행 방식 선택 요청
- 병렬 응답 또는 토론 모드 선택
- 아이디어 클러스터링
- 최종 요약
- 핵심 출력: 아이디어 목록, 그룹화, 추천안

## Recommended Product Strategy

### 원칙
- 제품은 하나의 명확한 진입점을 가져야 한다.
- 엔진은 처음부터 재사용 가능하게 설계해야 한다.
- 브랜드 설명은 짧고 자극적이어야 한다.

### 권장 전략
1. 외부 제품은 `ssawar`라는 이름 아래 `AI 배틀` 하나로 출시
2. 내부 엔진은 `멀티 AI 오케스트레이터`로 설계
3. 배틀이 자리 잡으면 `실험실` 또는 `게임` 탭으로 다른 모드 추가

이 전략이 좋은 이유는 다음과 같다.
- 마케팅 메시지가 흐려지지 않는다.
- 사용자에게는 단순한 제품처럼 보인다.
- 내부 팀은 미래 확장을 위해 다시 뜯어고칠 필요가 줄어든다.

### 제품 설명 예시
- 바깥 설명: `AI들끼리 싸움을 붙여보는 서비스`
- 투자자/팀 내부 설명: `멀티 AI 오케스트레이션 커널 위에 배틀 경험을 얹은 소비자 서비스`

## MVP Engine Scope

MVP 엔진은 아래 공통 기능까지만 지원하면 충분하다.

1. 오케스트레이터 모델 등록
2. 참가자 등록
3. 역할 부여
4. 순차/동시 실행 선택
5. 페이즈 전환
6. 상태 저장
7. 프롬프트 생성
8. 응답 정규화
9. 종료 조건 판정
10. 결과 저장

### Battle-First MVP Product Scope
- 유저는 세션 생성 시 오케스트레이터 AI 모델을 선택한다.
- 유저는 최대 16개의 참가 AI를 등록할 수 있다.
- 첫 공개 경험은 `배틀 모드` 하나만 노출한다.
- 세션 톤과 진행 스타일은 유저가 직접 세세하게 고르기보다 오케스트레이터가 정하거나 필요한 경우 짧게 확인한다.
- 배틀 MVP에서는 유저 투표와 공식 승자 판정을 두지 않는다.
- 엔진은 내부적으로 다른 모드 확장이 가능해야 하지만, UI는 배틀 경험에 집중한다.

### MVP에서 제외할 것
- 복잡한 비밀 정보 시스템
- 실시간 음성 모드
- 장기 메모리
- 인간 다중 참여 실시간 동기화
- 사용자 정의 규칙 DSL

## State Model

### Common Session State
- session_id
- mode_type
- orchestrator_model
- phase
- participants
- public_messages
- private_messages
- scores
- timer_state
- stop_reason

### Public vs Private State
- 배틀은 거의 대부분 public state로 충분하다.
- 마피아 같은 게임은 participant별 private state가 필요하다.
- 따라서 엔진은 처음부터 public/private state 분리를 지원하는 편이 낫다.

## Orchestrator Decision Design

### 규칙 기반으로 처리할 것
- 턴 순서
- 최대 턴 수
- 타임아웃
- 점수 합산
- 단계 전환
- 참가 AI 최대 16 제한

### LLM 판단을 허용할 것
- 어떤 반박이 더 날카로운가
- 어떤 아이디어가 새로운가
- 현재 논의가 반복적인가
- 요약 시 어떤 쟁점이 핵심인가

### 혼합형이 좋은 이유
- 전부 LLM에 맡기면 비용과 일관성 문제가 생긴다.
- 전부 규칙 기반이면 재미와 유연성이 떨어진다.

## Risks and Mitigations

### 리스크 1. 너무 범용적으로 시작하면 아무 것도 날카롭지 않음
- 대응: 출시 경험은 반드시 하나로 고정한다. 현재는 배틀만 출시한다.

### 리스크 2. 엔진 추상화가 과해서 초기 개발 속도가 느려짐
- 대응: 2~3개 모드에 공통인 추상화만 먼저 만든다. 미래의 모든 경우를 일반화하지 않는다.

### 리스크 3. 마피아 같은 모드는 private state가 복잡함
- 대응: private state가 필요한 모드는 배틀 출시 이후 두 번째 확장 단계로 미룬다.

### 리스크 4. 심판 역할을 LLM에 맡기면 판정 일관성이 떨어짐
- 대응: 판정은 가급적 규칙 기반으로, 애매한 부분만 LLM 보조 판단으로 둔다.

## ADR

### Decision
서비스는 `배틀 단일 제품`으로 출시하되, 내부 구조는 `범용 멀티 AI 오케스트레이터 엔진`으로 설계한다.

### Drivers
- 배틀은 초기 진입이 쉽다.
- 엔진 재사용성은 장기 확장성을 만든다.
- 코드 중복을 줄이고 모드 추가 속도를 높일 수 있다.

### Why Chosen
- 사용자에게는 단순한 진입점이 필요하다.
- 팀 내부에는 재사용 가능한 커널이 필요하다.
- 이 둘을 분리하면 초기 집중과 장기 확장을 동시에 가져갈 수 있다.

### Consequences
- MVP에서는 일부 추상화 비용을 감수해야 한다.
- 대신 다음 모드 추가 비용이 크게 내려간다.
- 제품 메시지는 배틀 중심으로 유지해야 한다.

### Follow-ups
- 배틀 커널에서 어떤 추상화가 실제로 재사용되는지 검증
- brainstorm mode를 두 번째 후보로 평가
- private state를 위한 데이터 모델 별도 설계
