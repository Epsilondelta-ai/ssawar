# AI Battle 서비스 MVP 기획안

작성일: 2025-02-14
상태: Draft v1

## Brand Definition

### 프로젝트명
- `ssawar`
- 한글 발음은 `싸워`

### 네이밍 의미
- 여러 AI를 한 자리에 모아 `싸움을 붙인다`는 제품 인상을 직접적으로 전달한다.
- 짧고 기억하기 쉽고, 밈화/공유에 유리하다.

### 도메인
- `ssawar.ai`

### 대외 브랜드 메시지
- `AI들끼리 싸움을 붙여보는 곳`
- 표면적으로는 멀티 AI 엔터테인먼트 서비스로 인식되게 한다.

### 내부 제품 전략
- 외부 메시지는 `싸움`, `배틀`, `붙여보기`에 집중한다.
- 내부 구조는 배틀 전용 서비스가 아니라 범용 오케스트레이터 엔진으로 설계한다.
- 즉, 브랜드는 날카롭게 하나로 세우고 엔진은 범용적으로 만든다.

## Requirements Summary

### 목표
여러 AI를 한 채팅방에 모아 같은 주제에 대해 답변하게 하고, 서로 반박하거나 보완하게 만드는 `AI 배틀` 서비스를 만든다.

### 핵심 타깃
- 일반 사용자
- AI 성능 비교보다 `재미`, `구경거리`, `참여감`이 더 중요한 사용자

### 핵심 가치 제안
- 하나의 질문으로 여러 AI의 성격 차이와 답변 스타일 차이를 즉시 체감할 수 있다.
- 유저는 단순히 답을 받는 것이 아니라 `AI들끼리 붙는 장면`을 소비하고 개입할 수 있다.
- 결과적으로 `재미형 엔터테인먼트`와 `가벼운 정보 탐색`을 동시에 제공한다.

### 문제 정의
- 현재 대부분의 AI 서비스는 1:1 대화 구조라 비교 감상이 어렵다.
- 여러 AI를 번갈아 쓰려면 탭 전환, 복사/붙여넣기, 수동 비교가 필요하다.
- 유저는 "누가 더 설득력 있는가", "누가 더 웃기게 답하는가", "누가 더 논리적인가"를 한 장면 안에서 보고 싶어 한다.

### MVP 범위
- 유저가 `채팅 세션 생성 시` 오케스트레이터 AI 모델을 선택
- 유저가 참가 AI 목록을 고르고, 세션당 최대 16개까지 등록 가능
- 오케스트레이터가 세션 톤, 발언 순서, 개입 시점, 종료 타이밍을 자율 결정하거나 필요시 유저에게 질문
- AI 간 반박/재반박 및 자발적 참전
- 동적 연장 또는 종료
- 세션 요약 및 하이라이트 생성
- 세션 로그 저장 및 공유

### 비범위
- 업무용 취합 모드
- 문서 생산성 중심 UI
- 팀 협업 기능
- 장기 메모리 기반 개인화
- 복잡한 에이전트 워크플로 자동화
- 마피아 같은 private-state 게임 모드
- 유저 정의 커스텀 규칙 엔진

## Product Definition

### 서비스 한 줄 설명
`한 질문으로 여러 AI를 붙여 보고, 읽고, 지켜보는 AI 배틀 아레나`

### 제품 포지셔닝 업데이트
- 대외적으로는 `ssawar`를 `AI 배틀` 단일 경험으로 출시한다.
- 내부적으로는 배틀 전용 하드코딩이 아니라 `범용 오케스트레이터 엔진` 위에 배틀 모드를 첫 번째 템플릿으로 올린다.
- 이후 마피아, 끝말잇기, 동시 응답 비교, 브레인스토밍 같은 모드는 같은 엔진 위에 추가한다.

### 추천 태그라인
- `AI들끼리 싸움을 붙여보자`
- `누가 더 잘났는지 지금 붙여봐`
- `질문 하나, AI 여러 명, 결과는 한 판 승부`

### 핵심 컨셉
유저가 주제를 던지면 GPT, Claude, Gemini 같은 AI들이 같은 방에서 각자 주장하고, 서로 반박하며, 마지막에는 세션 요약과 하이라이트가 남는 구조다.

### 기대 사용자 행동
1. 유저가 새 채팅 세션을 만든다.
2. 시스템이 `Untitled` 상태의 빈 세션 화면을 연다.
3. 유저가 첫 메시지를 보내면 오케스트레이터가 진행 방식, 톤, 순서를 자율 결정하거나 필요시 유저에게 짧게 묻는다.
4. AI들이 서로를 인용, 반박, 편들기 하며 배틀이 진행된다.
5. 유저는 구경만 하거나, 중간에 짧게 개입하거나, 채팅을 계속 이어간다.
6. 세션 제목은 대화 내용으로 자동 생성되고, 유저는 원하면 수정한다.
7. 충분히 봤다고 느끼면 세션을 나가거나 결과를 공유한다.
8. 재미있는 세션 로그가 바이럴 콘텐츠로 재소비된다.

## Core User Scenarios

### 시나리오 A: 재미형 대결
- 유저가 "누가 더 똑똑한지 싸워봐" 같은 주제를 던진다.
- AI들이 각자 주장하고 서로 공격한다.
- 유저는 한동안 구경하다가, 더 붙게 하거나 그냥 세션을 나간다.

### 시나리오 B: 팬덤형 비교
- 유저가 특정 모델을 응원하며 배틀을 연다.
- 배틀이 재미있으면 결과를 공유하고, 아니면 그대로 떠난다.

### 시나리오 C: 가벼운 정보형 대결
- 유저가 "2025년에 유튜브 시작해도 늦지 않았나?" 같은 질문을 던진다.
- AI별 다른 관점이 드러나고, 유저는 더 물어보거나 만족하면 세션을 종료한다.

## MVP Feature Set

### 1. 채팅 세션 생성
- 오케스트레이터 AI 모델 선택
- 참가 AI 선택: 최소 2개, 최대 16개
- 공개 범위 선택: 비공개 / 링크 공유 / 공개
- 세션은 기본 제목 `Untitled`로 생성

### 1-0. 생성 시 책임 분리
- `유저`는 오케스트레이터 모델, 참가 AI 목록, 공개 범위를 정한다.
- `오케스트레이터`는 세션의 톤, 발언 순서, 개입 시점, 마무리 시점을 결정한다.
- 필요하면 오케스트레이터가 유저에게 진행 강도나 분위기를 짧게 확인할 수 있다.
- 세션 제목은 생성 시 `Untitled`이며, 이후 대화 내용으로 자동 생성되고 유저가 수정할 수 있다.

### 1-1. 기본 파라미터
- 기본 참가 AI 수: 2
- 최대 참가 AI 수: 16
- 기본 오케스트레이터 모델: 유저가 세션 생성 시 선택
- 기본 최대 턴 수: 6
- 기본 응답 길이: 턴당 120~220자 또는 1~3문단
- 기본 지연 경고 시간: 참가 AI 12초, 오케스트레이터 20초
- 기본 hard timeout: 참가 AI 45초, 오케스트레이터 90초
- 기본 종료 방식: 자동 연장 + 반복 감지 종료

### 1-2. 응답 시간 정책
- `지연 경고 시간`은 응답이 늦다고 느껴지기 시작하는 UX 기준값이다.
- `hard timeout`은 요청을 끊거나 우회 처리할 수 있는 시스템 기준값이다.
- 참가 AI는 상대적으로 짧게, 오케스트레이터는 더 길게 허용한다.
- 고지능 모델이나 장문 응답은 40초를 넘길 수 있으므로, hard timeout은 보수적으로 잡는다.

### 2. 턴 엔진
- 시작 턴에서는 모든 AI가 자신의 첫 입장을 낸다.
- 이후 턴에서는 특정 AI의 발언을 인용해 반박, 동조, 재정리할 수 있다.
- 시스템은 `긴장도`, `새 정보량`, `반박 강도`, `반복도`를 기준으로 다음 턴 지속 여부를 판단한다.
- 유저는 언제든 `더 붙여`, `이제 끝내`, `최종 정리해` 같은 개입을 할 수 있다.
- 각 응답은 길이 제한과 시간 제한 규칙을 가진다.

### 2-0. 턴 구조
1. 시스템이 현재까지의 핵심 발언을 요약한다.
2. 다음 턴 참여 AI를 선정한다.
3. 선택된 AI는 특정 상대 발언을 인용하거나 전체 흐름에 대한 입장을 낸다.
4. 시스템은 새 주장 여부와 배틀 열기를 평가한다.
5. 종료 또는 다음 턴 진입을 결정한다.

### 2-1. 다음 턴 참여자 선정 규칙
- 기본값은 `직전 턴에서 가장 직접적으로 공격받은 AI`를 우선 호출한다.
- 새 근거를 먼저 낸 AI가 있으면 그 주장에 대해 반박 가능한 AI를 우선 호출한다.
- 3개 이상 AI가 있을 경우, 같은 AI가 2턴 연속 독점하지 않도록 가중치를 낮춘다.
- 특정 AI가 2턴 연속 실질 발언 없이 반복하면 우선순위를 내린다.
- 유저가 특정 AI를 지목해 `너 답해봐` 형태로 호출할 수 있다.

### 2-2. 종료 조건
- 최대 턴 수 도달
- 새 주장 없이 반복이 이어짐
- 점수 변화가 거의 없음
- 유저가 종료 선택
- 비용 또는 시간 한도 도달

### 2-3. 연장 조건
- 직접적인 상호 반박이 발생함
- 새로운 근거 또는 예시가 제시됨
- 유저 반응 또는 개입이 큼
- 특정 AI가 강한 공격을 받았고 재응답 가치가 큼

### 2-4. 연장/종료 판정 점수
- 시스템은 매 턴 종료 시 아래 항목을 0~2점으로 평가한다.
- `반박 강도`: 특정 주장에 직접 반론이 붙었는가
- `새 정보량`: 이전 턴에 없던 근거, 예시, 프레임이 나왔는가
- `관전 재미`: 문장 밀도, 유머, 도발, 명확성이 살아 있는가
- `반복 패널티`: 같은 논지 반복, 의미 없는 감정적 발화가 늘었는가
- 총점이 높으면 연장, 낮으면 종료 후보로 둔다.

### 2-5. MVP용 간단 종료 정책
- 총점 `4점 이상`: 다음 턴 자동 진행
- 총점 `2~3점`: 유저에게 `더 붙일까?` 선택 노출
- 총점 `1점 이하`: 시스템이 마무리 요약 단계로 이동
- 단, 최대 턴 수에 도달하면 점수와 무관하게 종료

### 2-6. 마무리 단계
- 종료 결정 후 각 AI에게 `최종 한줄` 대신 `최종 입장 정리` 기회를 1회 줄 수 있다.
- 이 단계는 배틀이 아니라 `정리 단계`로 분리해 톤을 낮춘다.
- 결과 화면에는 배틀 중 발언과 정리 발언을 구분 표시한다.

### 3. AI 페르소나/역할 레이어
- 같은 모델이라도 `논리파`, `독설가`, `중재자`, `개그형` 같은 배틀 캐릭터 부여 가능
- 초기 MVP에서는 실제 모델 특성보다 `프롬프트 페르소나` 중심으로 차별화
- 페르소나와 톤은 유저가 직접 고르기보다 오케스트레이터가 세션 맥락에 맞게 결정하거나 확인 질문 후 정한다.

### 4. 관전 UX
- 카드형 답변 비교
- 턴별 타임라인
- 반박 관계 시각화
- 현재 중심 발화 AI 표시

### 4-1. 라이브 배틀 조작 버튼
- `더 붙여`: 자동으로 다음 핵심 반박 턴 진행
- `이 AI 답변`: 특정 AI를 강제로 다음 턴에 호출
- `주제 비틀기`: 주제의 하위 쟁점을 새로 던짐
- `이제 정리`: 즉시 마무리 단계로 전환
- `종료`: 현재 상태로 결과 계산

### 4-2. 라이브 시각 요소
- 현재 발화 AI 강조
- 누구를 겨냥한 반박인지 화살표 표시
- 열기 게이지: 지금 배틀이 살아 있는지 직관적으로 보여줌
- 반복 경고: 비슷한 말이 반복되면 UI에 약하게 표시

### 5. 결과 및 정리
- 세션 종료 후 핵심 쟁점 요약
- 하이라이트 발언 3~5개 추출
- 세션 성격 태그: 논리적, 창의적, 웃김, 공격적, 반복적

### 5-1. 결과 산정 방식
- 시스템은 공식 우열을 강제하지 않는다.
- 결과는 `핵심 쟁점`, `강한 발언`, `세션 흐름`, `종료 사유` 중심으로 정리한다.
- 필요하면 오케스트레이터가 `어떤 발언이 가장 강하게 보였는지`를 참고 코멘트로 남길 수 있지만 공식 판정으로 취급하지 않는다.

### 5-2. 결과 화면 구성
- 배틀 요약 3줄
- 하이라이트 발언 3~5개
- 턴 흐름 타임라인
- 재배틀 버튼
- 공유용 짧은 링크와 이미지 카드

### 6. 공유
- 배틀 결과 링크 생성
- 결과 요약 카드 생성
- "이 주제로 다시 붙이기" CTA 제공

## Product Principles

1. 첫 30초 안에 재미가 보여야 한다.
2. 비교보다 `드라마`가 먼저 느껴져야 한다.
3. 복잡한 설정 없이 바로 배틀을 열 수 있어야 한다.
4. 모델 품질 차이보다 `경험 연출`이 MVP 완성도를 좌우한다.
5. 유저 생성 콘텐츠가 자연스럽게 공유돼야 한다.

## UX / IA

### 핵심 화면
- 홈
- 채팅 세션 생성
- 라이브 세션
- 결과 화면
- 공유된 세션 보기

### 홈 화면 구성
- `새 세션` CTA
- 인기 배틀 주제
- 오늘의 명승부
- 추천 템플릿: "누가 더 창업 잘할까", "누가 더 설득력 있나"

### 라이브 세션 UI
- 상단: 주제, 라운드 상태, 남은 턴
- 중앙: AI별 카드
- 하단: 유저 액션 영역
- 우측 또는 모바일 하단: 공유, 재배틀, 세션 제어

### 라이브 세션 상세 정보 구조
- 상단 바: 세션 제목, 현재 턴, 남은 최대 턴
- 상단 바 보조 정보: 선택된 오케스트레이터 모델
- 메인 패널: 현재 발화, 직전 발화, 반박 관계
- 보조 패널: 참가 AI 소개, 현재 열기, 유저 실시간 반응
- 액션 바: 더 붙여, 특정 AI 호출, 정리, 종료

## Differentiation

### 기존 AI 챗과의 차별점
- 1:1 대화가 아니라 `멀티 AI 대결`이 기본 구조
- 정답 생산보다 `구경 가능한 상호작용`이 핵심
- 배틀 로그 자체가 콘텐츠가 된다.

### 브랜드 차별점
- 대부분의 멀티 AI 제품은 비교, 생산성, 분석 톤으로 간다.
- `ssawar`는 처음부터 `싸움을 붙인다`는 감정적이고 대중적인 프레이밍을 취한다.
- 이 덕분에 일반 사용자는 제품 용도를 즉시 이해할 수 있다.

### 방어 가능한 포인트
- 모델 연결 자체는 복제 가능하지만, 배틀 형식/연출/요약/공유 루프는 제품 경험 자산이 된다.
- 인기 주제, 세션 로그, 캐릭터성, 배틀 포맷이 누적될수록 네트워크 효과가 생긴다.

## Acceptance Criteria

1. 유저는 1분 이내에 첫 채팅 세션을 생성할 수 있어야 한다.
2. 유저는 세션 생성 시 오케스트레이터 AI 모델을 직접 선택할 수 있어야 한다.
3. 하나의 채팅 세션에는 최대 16개의 AI를 참가시킬 수 있어야 한다.
4. 최소 참가 AI 수는 2여야 한다.
5. 세션 생성 시 질문은 필수가 아니어야 한다.
6. 세션 생성 직후 화면은 `Untitled` 제목의 빈 채팅 화면이어야 한다.
7. 세션 제목은 대화 내용으로 자동 생성돼야 하며 유저가 수정할 수 있어야 한다.
8. 시스템은 고정 라운드가 아니라 동적 턴 기반 배틀 구조를 지원해야 한다.
9. 시스템은 턴별로 `연장`, `유저 확인`, `종료` 중 하나를 선택할 수 있어야 한다.
10. 유저는 진행 중 `더 붙여`, `이제 정리`, `종료`, `특정 AI 호출`을 실행할 수 있어야 한다.
11. 결과 화면에는 핵심 발언 요약, 하이라이트, 종료 사유가 표시돼야 한다.
12. 각 세션은 고유 링크로 공유 가능해야 한다.
13. 모바일에서도 핵심 기능인 생성, 관전, 공유가 가능해야 한다.
14. 서비스는 유해 발언, 증오 표현, 과도한 공격 유도에 대해 기본 안전장치를 가져야 한다.
15. 시스템은 기본 무료 플랜에서 세션당 최대 턴 수와 응답 길이를 제한할 수 있어야 한다.

## Technical Approach

### 권장 아키텍처
- 프론트엔드: 웹 우선, 모바일 반응형
- 백엔드: 배틀 세션 오케스트레이션 API
- 모델 어댑터 레이어: OpenAI / Anthropic / Google 공급자 분리
- 저장소: 배틀 메타데이터, 메시지 로그, 세션 요약 저장
- 비동기 처리: 턴 생성 및 재시도 큐

### 아키텍처 원칙
- 배틀 규칙은 엔진 코드에 직접 박지 않고 `mode template`으로 분리한다.
- 엔진은 `참가자`, `순서`, `페이즈`, `액션`, `종료 조건`만 공통으로 다룬다.
- 배틀은 그 위에서 `반박 중심 턴 규칙`을 가진 하나의 모드로 구현한다.

### 핵심 기술 포인트
- 모델별 API 응답 포맷 표준화
- 턴 기반 orchestration engine
- 턴 연장/종료 판정 로직
- 참가자 선택 가중치 로직
- 토큰 비용 제한 장치
- 응답 지연 시 fallback 또는 타임아웃 처리
- unsafe output moderation layer

### 오케스트레이션 규칙 초안
- 모든 모델 출력은 내부 공통 포맷으로 정규화한다.
- 오케스트레이터 모델은 참가 AI와 별도 슬롯으로 취급한다.
- 각 턴 프롬프트에는 `현재 주제`, `직전 핵심 발언`, `반박 대상`, `톤 규칙`, `길이 제한`만 넣는다.
- 톤 규칙은 유저가 직접 고정하지 않고, 오케스트레이터가 세션 문맥상 결정하거나 유저에게 짧게 재질문한다.
- 전체 로그를 매 턴 다 주지 않고, 최근 핵심 3~5개만 요약해 넣는다.
- 반복 발화가 감지되면 프롬프트에 `새 근거 없으면 짧게 정리하라`를 추가한다.
- 종료 직전에는 `더 싸우기`보다 `핵심 쟁점 정리`를 유도한다.
- 참가 AI가 지연 경고 시간을 넘기면 UI에 `생각 중` 상태를 표시한다.
- 참가 AI가 hard timeout을 넘기면 해당 턴은 건너뛰거나 짧은 fallback 멘트로 대체할 수 있다.
- 오케스트레이터가 hard timeout을 넘기면 세션은 실패시키기보다 `계속 생성 중` 또는 백그라운드 처리 상태로 넘기는 편이 낫다.

### 비용 제어 정책
- 무료 유저는 기본 2개 참가 AI, 최대 6턴
- 프리미엄 유저는 최대 16개 참가 AI, 최대 12턴
- 각 턴 최대 토큰 상한을 둔다.
- 공개 인기 배틀은 요약 캐시를 재사용해 비용을 줄인다.
- 응답이 늦는 모델은 한 턴 건너뛰게 할 수 있다.
- 느린 고지능 모델은 세션 전체를 막지 않도록 timeout 이후 fallback 정책을 적용한다.

## Battle Flow Example

### 예시: "누가 더 설득력 있게 창업 조언을 하는가?"
1. 유저가 GPT와 Claude를 선택하고 `자동 연장` 모드로 배틀 시작
2. 시작 턴에서 두 AI가 각자 창업 조언을 제시
3. GPT가 Claude의 조언이 추상적이라고 공격
4. Claude가 구체 사례를 들며 재반박
5. 시스템이 새 정보와 반박 강도가 높다고 판단해 자동 연장
6. 세 번째 턴에서 두 AI가 비슷한 말만 반복
7. 시스템이 `더 붙일까?`를 띄움
8. 유저가 `이제 정리`를 누름
9. 각 AI가 최종 입장 정리를 1회 제출
10. 결과 화면에서 요약과 하이라이트를 확인한 뒤 링크 공유

## Risks and Mitigations

### 리스크 1. 실제로는 한두 번 써보고 끝날 수 있음
- 대응: 공유 중심 설계, 인기 주제 템플릿, 결과 카드 생성으로 재유입 루프 설계

### 리스크 2. 모델 비용이 빠르게 증가할 수 있음
- 대응: 무료 플랜에 턴 수/참여 모델 수 제한, 응답 길이 제한, 캐시 가능한 공개 배틀 활용

### 리스크 3. 모델 간 차이가 생각보다 재미없을 수 있음
- 대응: 모델 고유성에만 기대지 않고 배틀 캐릭터와 프롬프트 연출을 적극 사용

### 리스크 4. 독성 콘텐츠가 유도될 수 있음
- 대응: 주제 제한, 입력/출력 모더레이션, 신고 기능, 민감 카테고리 차단

### 리스크 5. API 장애나 지연으로 배틀 흐름이 끊길 수 있음
- 대응: soft timeout, hard timeout, 재시도, 응답 지연 표시, 대체 멘트 처리

### 리스크 6. 끝없이 이어져 피로감이 생길 수 있음
- 대응: 자동 종료 정책, 최대 턴 수, 반복 감지, `이제 정리` 버튼 제공

## ADR

### Decision
1차 제품은 `멀티 AI 범용 플랫폼`이 아니라 `일반 사용자 대상 AI 배틀 서비스`로 집중한다.

### Drivers
- 초기 차별점이 명확하다.
- 공유와 바이럴에 유리하다.
- 기능 범위를 줄여 출시 속도를 높일 수 있다.
- 여러 AI 동시 사용의 장점을 가장 직관적으로 보여준다.

### Why Chosen
- 배틀은 첫인상과 입소문 요소가 강하다.
- 일반 사용자 대상에서는 생산성보다 재미가 진입장벽이 낮다.
- 비교/취합/브레인스토밍은 추후 같은 멀티 AI 엔진 위에 확장 가능하다.

### Consequences
- 초기에는 엔터테인먼트 성격이 강해진다.
- 업무형 유저 전환은 후순위가 된다.
- 안전장치와 모더레이션 품질이 제품 신뢰를 좌우하게 된다.

### Follow-ups
- MVP 이후 비교 모드 확장 가능성 검토
- 크리에이터 모드 및 라이브 관전 기능 검토
